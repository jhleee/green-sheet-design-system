이 시스템에서 다크 표면은 딱 한 가지를 뜻한다 — 기계가 쓴 것. 사람이 승인하는 파싱 스펙과 LLM이 생성한 SQL이 전부다. 표면의 배경은 ink(#1B2730) 그 자체다. 종이에 잉크를 얹는 게 아니라, 잉크 위에 쓴다 — 라이트 UI를 뒤집은 것이 아니라 같은 잉크의 다른 면이다.

파싱 스펙 블록 LLM 제안 → 사람 승인 
# parsing_spec: 간편장부.yaml
layout: periods_in_rows
grain: month                     # 최소 단위 사실
header: {rows: [3, 4], type: two_level}
melt:
  upper_header → metric        # 수입/비용/고정자산증감
  lower_header → value_type    # 금액/부가세
exclude_rows:
  pattern: "(분기|반기|연) 합계"
  use_for: validation           # 버리되 checksum으로 활용
period_resolution:
  source: cell(I2)              # 과세년도 → "1월"을 2025-01로
  require_if_empty: user_input

3색만 쓴다 — 키워드 green-bright, 문자열 amber-code, 주석 code-dim. 라이트 UI의 의미색이 다크에서도 같은 뜻을 유지한다: green은 여전히 구조, amber는 여전히 리터럴 값이다.

생성된 SQL + 규칙 칩 TEXT → SQL
GENERATED SQL
SELECT period, metric, SUM(value) AS amount
FROM fact_financials
WHERE dataset = '간편장부'
  AND value_type = '금액'
  AND metric IN ('수입', '비용')
GROUP BY period, metric
ORDER BY period;
✓ metric alias: 매출→수입 ✓ 소계 미적재 → 이중집계 없음 ✓ flow 지표 → SUM 허용

규칙 칩은 SQL이 왜 그렇게 생성됐는지를 밝힌다. 결과만 보여주는 대신 근거를 남겨야 사람이 승인할 수 있다. green 계열 반투명이라 다크 위에서도 "검증됨"이라는 뜻이 그대로 유지된다.

스키마 덤프가 아니라 의미 계층이다. 동의어·집계 규칙·파생 수식이 LLM의 SQL 생성 오류를 줄인다. 카드의 배지 색이 곧 그 지표의 성격이다 — green은 그냥 더하면 되는 flow, purple은 파생, amber는 조심해서 다뤄야 하는 것.

차트에는 전용 토큰이 없다. 시리즈는 green → amber → purple 순서로 의미 팔레트를 그대로 쓴다 — 차트가 나머지 UI와 같은 언어를 말하게 하기 위함이다 (chart-1/2/3 별칭은 green/amber/purple의 순수 중복이라 제거했다). 그리드라인도 표의 셀 격자와 같은 line-soft이고, 축 눈금은 보조 텍스트와 같은 ink-soft다. 원본은 y축(#8A98A2)과 x축(#5C6E7A)에 서로 다른 회색을 쓰고 있었다 — 이제 통일됐고, 흰 배경 대비도 2.96:1 → 5.29:1로 올라 WCAG AA(4.5:1)를 넘긴다. YTD 라벨도 같은 이유로 amber-ink가 되어 3.13:1 → 5.37:1이 됐다. 둘 다 원래는 본문 크기에서 AA 미달이었다. YTD 같은 신뢰도 플래그는 색이 아니라 투명도 + 점선 테두리 + 라벨로 표시한다. 색만으로 경고하면 색각 이상 사용자에게는 경고가 전달되지 않는다. SVG 안의 색은 유틸리티가 닿지 않으므로 리터럴 hex를 쓰되, 값은 전부 토큰과 동일하다.

규칙 — 다크 표면은 코드에만 쓴다. 시각적 강조가 필요하다는 이유로 일반 UI를 다크로 만들지 않는다. 그러면 "기계가 쓴 것"이라는 신호가 희석되고, 다크 표면은 그냥 또 하나의 장식이 된다.
